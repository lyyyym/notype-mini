## 2. 关键决策回顾

### 2.1 气泡：从"圆形黑底"改成"白色卡片 + 黑色胶囊"

**最初的判断**：用户说"气泡好难看，有一圈白色边缘，而且声纹不会跳动"，我以为是 CSS 样式问题。

**实际原因**：三层透明都没做好：
- CSS 全局 `body { background: #f5f5f7; }` 会透过 transparent Tauri 窗口显成灰白边缘。
- Tauri 窗口默认带阴影，透明窗口 + 阴影 = 白色光晕。
- NSWindow 本身默认 opaque，需要显式 `setOpaque: false`、`setBackgroundColor: clearColor`、`setHasShadow: false`。

**学到的**：Tauri 的 `transparent: true` 不是魔法，macOS 透明窗口要 CSS + 配置 + NSWindow 三层配合。参考项目（dandandujie/notype）的 `html, body { background: transparent; }` 是关键第一步，但还不够。

**结果**：气泡现在跟随鼠标，白色卡片 + 黑色胶囊 + 9 柱 RMS 声纹，无白色边缘。

### 2.2 词典"没效果"：不是 bug，是反馈缺失

**用户反馈**："我加入了一些容易纠错的词，但好像并没有看到效果。"

**排查过程**：日志显示 `词典替换后` 和 `识别结果` 完全一样。查配置文件发现 `dictionary = []`——词条根本没保存。

**根因**：添加/删除词条只改前端状态，必须点"保存配置"才落盘。但界面没有任何提示告诉用户这一点。

**学到的**：
- "没效果"的反馈，50% 以上是用户操作流程与产品心智模型不一致，而不是代码 bug。
- 对于"编辑 → 保存"两段式操作，必须给出明确的 dirty 状态反馈。
- 我最初的实现让"保存配置"按钮藏在 DashScope 卡片里，和词典卡片物理距离很远，进一步加剧了误解。

**修复**：
- 添加/删除后状态栏即时提示"保存配置后才会生效"。
- 词典卡片顶部增加橙色警告条 +「立即保存」按钮，直接在操作现场给出反馈。
- 保存成功后警告条消失。

### 2.3 历史记录滚动：小问题，大影响

**用户反馈**："历史记录这里可以做一个列表，否则一直往下堆叠会导致页面滑动过长。"

**决策**：直接给 `.history-list` 加 `max-height: 320px` + `overflow-y: auto`。简单，但显著改善主窗口可用性。

**学到的**：用户不会明确说"我需要一个固定高度的滚动容器"，但会描述症状。把症状翻译成最小可行方案，比过度设计更重要。

### 2.4 自动学习词条：从"自动 diff"退回到"用户确认"

**最初的想法**：diff 历史记录里的 `raw_text` vs `polished_text`，自动发现候选词条。

** grilling 后意识到的致命问题**：`polished_text` 是 LLM 润色后的结果，不是用户纠正。自动 diff 会学到一堆 LLM 的坏习惯：
- `嗯 → 空`
- `那个 → 空`
- `我昨天——不，前天 → 我前天`

这些不是识别错误，是口语整理规则。自动加入词典会污染整个替换系统。

**修正方向**：必须拿到**用户的真实纠正行为**。具体方案：
- 在历史记录（仅录音类型）加「修正」按钮。
- 用户把 `polished_text` 改成正确版本后保存。
- 后端 diff **ASR 原文 `raw_text`** 与 **用户修正版本**，提取候选。
- 候选进入"待确认"队列，用户一键接受/拒绝。
- 接受后进入正式词典，但**仍然走手动保存流程**（复用 `dictDirty` 警告条）。

**学到的**：
- "自动化学习"听起来很高级，但如果学习信号是错的，就是高级垃圾。
- 用户纠正行为是高质量信号，但必须通过 UI 显式捕获。
- 候选阶段保留人工确认，是精度与自动化之间的务实平衡。

## 3. 判断对了什么 / 判断错了什么

### 判断对了

- **enigo 0.3 升级失败后立即回滚**：当时 V3.1 尝试升级 enigo 到 0.3，普通转写直接崩溃。按预先约定的"验证失败即回滚"策略，revert 到 0.2 + `Key::Other` workaround，没有恋战。
- **编辑模式不显示气泡**：编辑需要操作目标应用焦点，气泡窗口再轻量也可能干扰。隐藏气泡是正确的。
- **shortcut 注册只注册变化的部分**：V3 早期保存配置会重新注册当前快捷键，导致 `RegisterEventHotKey failed` 并让整个保存失败。改成"比较新旧 → 只注册变化 → 成功后注销旧"，稳定下来。

### 判断错了 / 低估了

- **低估了"保存"语义的用户理解成本**：我以为"添加词条 + 点保存配置"是自然的，但用户完全没意识到这两步是分离的。以后凡是"编辑态 → 持久态"的转换，都要在 UI 上显式标出来。
- **低估了 Tauri 透明窗口的复杂度**：原以为 `transparent: true` 就够了，实际需要 CSS + window config + NSWindow 三层。以后遇到"透明但看到边缘"的问题，直接按三层清单排查。
- **最初高估了 raw vs polished diff 的价值**：grilling 前我真考虑过用它来自动学习，幸亏被 Codex 和用户纠正拉回来了。

## 6. 总体感受

本次迭代最大的价值不在功能本身，而在**把"用户没保存"和"LLM 习惯"两个隐藏假设显性化了**。产品稳定后，真正的瓶颈往往不是技术实现，而是：

- 用户是否清楚当前操作的状态？
- 学习信号是否真的来自用户意图？

这两个问题搞清楚之后，代码实现反而变得直接。
